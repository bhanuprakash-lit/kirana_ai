# Kirana AI — Developer Guide

> Audience: any developer joining the **kirana_ai** Flutter application. This document
> describes how the codebase is laid out, how a screen talks to the backend, the
> naming and state-management conventions in use, and the exact steps to add a new
> feature without breaking existing ones.
>
> The user-facing feature walkthrough lives in [`FEATURES.md`](FEATURES.md) and
> the step-by-step user manual in [`USER_MANUAL.md`](USER_MANUAL.md). The backend
> HTTP contract lives in `API_REFERENCE.md` inside the `kirana-master-backend`
> repo. This guide is strictly about the **mobile codebase**.

---

## 1. What this app is

Kirana AI is a cross-platform Flutter mobile app for South Indian kirana (retail)
store owners, published by LohiyaAI. It is a thin client over a custom FastAPI +
PostgreSQL backend (`kirana-master-backend`). The app does not run any business
logic locally — it stores authentication tokens, caches the last known user
profile, and otherwise reads/writes via REST.

- **Platforms:** Android (primary) and iOS. No web build in production.
- **Distribution:** Google Play (in-app purchases enabled). iOS Test Flight only.
- **Auth model:** custom Bearer token issued by the backend (`/kirana/auth/*`)
  plus a second JWT scoped to the POS module (`/pos/token`).
- **Firebase** is used **only** for Remote Config (dynamic base URL, kill switch,
  trial duration), Phone OTP, Crashlytics, Performance, and FCM push. We do **not**
  use Firebase Auth as the primary identity, Firestore, or Firebase Storage.

### Files in `lib/` root

`lib/brand_theme.dart`, `lib/home_page.dart`, and `lib/app_config.dart` are
**legacy scaffolding** from earlier iterations. They are not imported anywhere
(verified by grepping the codebase). The canonical equivalents are under
`lib/core/`. Treat them as safe to delete in a future cleanup, but do not change
them in passing.

### What goes in each per-feature folder

| Subfolder        | Purpose                                                                                             |
| ---------------- | --------------------------------------------------------------------------------------------------- |
| `models/`        | Plain Dart classes for API payloads (`fromJson` factories). Immutable, no Flutter imports.          |
| `providers/`     | Riverpod providers + Notifier classes that own the feature's runtime state.                        |
| `repositories/`  | Optional. Pure HTTP wrappers when a feature has more than ~3 endpoints (e.g. `auth`).               |
| `services/`      | Optional. Side-effectful helpers that are not strictly HTTP (e.g. Firebase OTP flow).               |
| `views/`         | Screens (one widget per file) + sub-folders `tabs/`, `widgets/`, `intelligence_details/` etc.       |
| `widgets/`       | Local widgets only used by this feature.                                                            |

## 4. Build, run, and tooling

### Prerequisites

- Flutter SDK matching `pubspec.yaml`'s `environment.sdk: ^3.11.4`
- Android Studio (with the Flutter plugin) **or** VS Code with the Dart + Flutter
  extensions
- A Firebase project — for OTP, FCM, Remote Config, Crashlytics, Performance

### First-time setup

```bash
flutter pub get
flutterfire configure          # regenerates lib/core/config/firebase_options.dart
```

You must enable **Phone Authentication** in the Firebase Console and register your
app's SHA-1 / SHA-256 fingerprints for Android, otherwise OTP login will fail.

### Day-to-day commands

```bash
flutter pub get                          # after pulling new dependencies
flutter analyze                          # static analysis (see §15)
flutter run                              # debug build on the connected device
flutter run --release                    # release build (uses Remote Config caching)
flutter build apk --release              # signed APK for sideloading / Play upload
flutter build appbundle --release        # Play Store upload artifact
flutter test                             # unit / widget tests (currently minimal)
```

### Backend URL

The base URL the app talks to is **not** hard-coded. At launch we read the
`backend_url` key from Firebase Remote Config. If Remote Config is unreachable we
fall back to a platform-appropriate default defined in
`lib/core/config/app_config.dart`:

| Platform        | Fallback                  |
| --------------- | ------------------------- |
| Android emulator | `http://10.0.2.2:9000`   |
| iOS / desktop   | `http://127.0.0.1:9000`   |
| Web             | `http://127.0.0.1:9000`   |

To point a local debug build at a remote server, set the `backend_url` parameter in
Firebase Remote Config (debug fetch interval is `Duration.zero`, so changes apply
on the next app launch).

### Other Remote Config keys

| Key                  | Type    | Default | Effect                                             |
| -------------------- | ------- | ------- | -------------------------------------------------- |
| `backend_url`        | string  | dev URL | Overrides the API base                             |
| `trial_days`         | int     | `14`    | Length of the trial granted by `/subscription/...` |
| `app_blocked`        | bool    | `false` | Hard-stops the app on the launch screen            |
| `blocked_reason`     | string  | `""`    | Message shown in the block screen                  |
| `blocked_store_ids`  | string  | `"[]"`  | JSON array of `store_id` integers to block         |

These are read inside `FirebaseBackendConfig._apply()` and pushed into static
fields on `AppConfig`. The block-state stream is consumed in
`SplashScreen` and `dashboard_screen.dart`.

## 5. Application entry & lifecycle

`main.dart`:

1. `WidgetsFlutterBinding.ensureInitialized()`
2. `FirebaseBackendConfig.initialize()` — fires Firebase, sets up Remote Config
   listeners, populates `AppConfig`.
3. Wires `FlutterError.onError` and `PlatformDispatcher.instance.onError` into
   Crashlytics. All uncaught exceptions (sync + async) reach Crashlytics.
4. Enables Firebase Performance collection.
5. Initialises local notifications (`initLocalNotifications()` from
   `support/providers/notification_provider.dart`).
6. Mounts `KiranaApp` inside a `ProviderScope`.

`KiranaApp` is a `ConsumerStatefulWidget` that also implements
`WidgetsBindingObserver` to capture foreground/background events and POST them to
`/kirana/tracking/app-event`. It builds a `MaterialApp.router` whose
`routerConfig` comes from `appRouterProvider` (see §7).

### Cold-start routing

`SplashScreen` is the initial route. After a 1.6 s logo animation it:

1. Checks `AppConfig.isAppBlocked` → routes to `AppBlockedScreen` if true.
2. Reads `onboarding_completed` from `SharedPreferences` → routes to `/onboarding`
   if false.
3. Checks `AppConfig.isStoreBlocked(storeId)` → store-level block screen if true.
4. Calls `AuthRepository.getToken()` — if a token exists and is not expired,
   navigates to `/home`; otherwise to `/login`.

### Navigation conventions

- Use `context.go('/path')` for **replacement** navigation (login → home, splash → login).
- Use `context.push('/path')` for **push** navigation when the user should be able
  to come back (FCM tap → screen).
- Pass complex parameters via the `extra` argument and **document the expected
  shape** in the route's `builder` — e.g. `state.extra as ReferralCampaign`.
- Do **not** call `Navigator.push` directly except in two places where it is
  intentional: `SplashScreen` → `AppBlockedScreen` (so it cannot be popped) and a
  handful of modal bottom sheets.

## 7. State management — Riverpod 3

### Hard rules

1. **Notifier API only.** Use `Notifier`, `AsyncNotifier`, or their `AutoDispose`
   variants. `StateNotifier`/`StateProvider` are intentionally **not used** in
   this codebase — they were removed during the Riverpod 3 upgrade. Do not
   reintroduce them.
2. **No `late final _repo` fields on a Notifier.** Riverpod 3 invalidates the
   instance when the provider is invalidated, and `late final` fields cause
   `LateInitializationError`. Read collaborators inside methods:
   `final client = ref.read(apiClientProvider);`.
3. **No state writes in `build()` synchronously.** If you need to kick off a load,
   wrap it in `Future.microtask(_load)` and return the initial state immediately.
   See `PosNotifier.build()` for the canonical example.
4. **Always expose the provider as a top-level `final`** named
   `<feature>Provider`. The Notifier subclass goes in the same file.

### The three patterns in use

#### a. Simple state Notifier — UI flags / scratch state

```dart
class TabNotifier extends Notifier<int> {
  @override
  int build() => 0;
  void switchTab(int index) => state = index;
}

final dashboardTabProvider =
    NotifierProvider<TabNotifier, int>(TabNotifier.new);
```

See `lib/features/dashboard/views/dashboard_screen.dart` for the tab providers.

#### b. AsyncNotifier — single resource that loads from the backend

This is the dominant pattern. The Notifier exposes a `_fetch()` that returns the
typed model, a `build()` that returns `_fetch()`, and a `refresh()` that resets to
`AsyncLoading` then `AsyncValue.guard`-wraps another `_fetch()`.

```dart
class OverviewNotifier extends AsyncNotifier<OverviewData> {
  @override
  Future<OverviewData> build() => _fetch();

  Future<void> refresh() async {
    state = const AsyncLoading();
    state = await AsyncValue.guard(_fetch);
  }

  Future<OverviewData> _fetch() async {
    final client = ref.read(apiClientProvider);
    // ... call /kirana/* and /pos/* in parallel ...
  }
}

final overviewProvider =
    AsyncNotifierProvider.autoDispose<OverviewNotifier, OverviewData>(
        OverviewNotifier.new);
```

Apply this pattern for any screen that fetches a single resource and supports
pull-to-refresh: `overviewProvider`, `financeProvider`, `procurementProvider`,
`inventoryProvider`, `customerProvider`, `subscriptionProvider`, `kpiProvider`,
`storeSettingsProvider`.

#### c. Notifier holding mutable composite state — POS-style screens

For a screen that holds many fields (cart, search, selected customer, loading
flags, error), define an **immutable state class** with a `copyWith`, and back it
with a `Notifier<MyState>`. The cart in `PosNotifier` (`lib/features/pos_inventory/
providers/pos_provider.dart`) is the model to copy.

The state class must implement `copyWith` with one named parameter per field plus
a `clear<X>` flag whenever a field is nullable and you need an explicit way to
clear it from the existing instance.

### Cross-provider invalidation

When a write affects another provider's data, call `ref.invalidate(thatProvider)`
inside the write method. Examples already in the code:

- `FinanceNotifier.addUdhaar` invalidates `customerProvider` because adding
  udhaar may create a new customer.
- `PosNotifier.placeOrder` calls `ref.read(overviewProvider.notifier).refresh()`
  after a successful order so the dashboard sales card updates.

Prefer **invalidate** over `refresh()` unless you specifically need to keep
in-flight state alive — e.g. `InventoryNotifier.refresh()` preserves the
optimistic `pendingItems` list.

## 8. Networking — `ApiClient`

`lib/core/services/api_client.dart` is the **only** place in the codebase that
talks HTTP directly (with the deliberate exception of `AuthRepository`, which
needs to call `/pos/token` with `application/x-www-form-urlencoded` before any
token exists). Every other feature must go through `ApiClient` via
`ref.read(apiClientProvider)`.

### Methods

| Method                                              | Auth token            | URL path                              | Decodes                       |
| --------------------------------------------------- | --------------------- | ------------------------------------- | ----------------------------- |
| `get(path)`                                         | Kirana                | `<base><path>`                        | `dynamic`                     |
| `post(path, body)`                                  | Kirana                | `<base><path>`                        | `dynamic`                     |
| `patch(path, body)`                                 | Kirana                | `<base><path>`                        | `dynamic`                     |
| `delete(path)`                                      | Kirana                | `<base><path>`                        | `dynamic`                     |
| `posGet(path)`                                      | POS                   | `<base><path>`                        | `Map<String, dynamic>?`       |
| `posGetList(path)`                                  | POS                   | `<base><path>`                        | `List<dynamic>?`              |
| `posPost(path, body)`                               | POS                   | `<base><path>`                        | `Map<String, dynamic>?`       |
| `getOltp(table, filters?)`                          | Kirana                | `<base>/oltp/<table>?<filters>`       | `Map` with `rows: List`       |
| `postOltp(table, body)`                             | Kirana                | `<base>/oltp/<table>`                 | `Map` with `row: Map`         |
| `patchOltp(table, body, filters?)`                  | Kirana                | `<base>/oltp/<table>?<filters>`       | `Map`                         |
| `deleteOltp(table, filters)`                        | Kirana                | `<base>/oltp/<table>?<filters>`       | `Map`                         |

Notes:

- **POS endpoints return `null`** if no POS token has been issued yet (e.g.
  phone-OTP user who has not exchanged for a POS token). Callers must handle this.
- **OLTP GET** wraps the result list under `'rows'`; **OLTP POST** wraps the new
  row under `'row'`. Mirror this when reading the response.
- **Non-2xx** responses throw `ApiException(statusCode, message)` defined in
  `auth/repositories/auth_repository.dart`. The message is extracted from the
  body's `error` or `detail` key when JSON; otherwise the raw body.

### Token storage and lifecycle

- Tokens live in `flutter_secure_storage`:
  - `auth_token` — the Kirana Bearer token from `/kirana/auth/{login,phone-login,
    register}`.
  - `pos_token` — the POS JWT from `/pos/token` (form-encoded user/pass) **or**
    `/pos/token-from-kirana` (phone-OTP users).
- Session metadata lives in `SharedPreferences`: `user_id`, `store_id`, `username`,
  `full_name`, `role`, `onboarding_completed`, `session_expiry`.
- `session_expiry` is set to `now + 30 days` on every login. `AuthRepository
  .getToken()` reads this first; if expired it clears both tokens and returns
  `null` (next call will land on `/login`).
- `clearSession()` does **not** remove `onboarding_completed` — the user is asked
  to log in again, not re-onboard.

| GET   | `/pos/products?store_id=…&limit=1000`               | `posProvider`, `inventoryProvider` |
| GET   | `/pos/products/barcode/{barcode}?store_id=…`        | `posProvider`           |
| GET   | `/pos/categories`                                   | `inventoryProvider`     |
| POST  | `/pos/orders`                                       | `posProvider`           |
| GET   | `/pos/orders?…`                                     | `posProvider` (history, today, customer)|
| GET   | `/pos/reports/daily-sales?date=…&store_id=…`        | `overviewProvider`      |

#### `/oltp/*` — generic CRUD (Kirana token)

The OLTP module is a generic table-level CRUD over `kirana_oltp.<table>`. Tables
the app reads/writes via `getOltp`/`postOltp`/`patchOltp`/`deleteOltp`:

`supplier`, `purchases`, `purchase_items`, `pricing`, `product`, `inventory`,
`inventory_batch`, `category`, `customer`, `khata`, `orders`.

> **Important schema quirks** when calling OLTP (these have already bitten us):
>
> - `product.barcode` is **globally unique** across stores. If `postOltp('product',
>   …)` raises a unique-constraint error on barcode, look the existing product up
>   via `getOltp('product', filters: {'barcode': barcode})` and reuse its
>   `product_id`. See `InventoryNotifier.addProduct` for the implementation.
> - `inventory` table has **no `reorder_level` column** — do not send one in
>   `postOltp('inventory', …)`.
> - `pricing.price` is nullable; `valid_from` is **NOT NULL** — always include it
>   on insert.
> - `GET /oltp/pricing` returns 422 if you pass `limit` — only send `store_id`.
> - `orders.user_id` is the FK to `kirana_oltp.users`, **not** `kirana_app_users`.

## 10. Conventions

### File naming

- All Dart files use **snake_case**. One public class per file. The file name is
  the snake_case form of the primary class — `OverviewNotifier` →
  `overview_provider.dart` (file holds the notifier **and** its provider).
- Screens end in `_screen.dart`. Modal sheets end in `_sheet.dart`. Card widgets
  end in `_card.dart`. Tabs go under a `tabs/` subfolder and end in `_tab.dart`.
- A feature's main barrel is `views/<feature>_screen.dart` even when it hosts
  sub-tabs (`pos_inventory_screen.dart`, `dashboard_screen.dart`).

### Class / type naming

- **Classes / enums / typedefs:** `UpperCamelCase`. (`OverviewData`, `PosState`.)
- **State classes** for composite Notifiers are named `<Feature>State` and live in
  the provider file. Always immutable with a `copyWith`.
- **Notifier classes** are named `<Feature>Notifier` and extend `Notifier<T>`,
  `AsyncNotifier<T>`, or their `AutoDispose` variants.
- **Providers** are top-level `final` named `<feature>Provider` (lowerCamelCase).
- **Model classes** match the backend JSON contract (`StoreInfo`,
  `RecommendationSummary`, `DailySales`, `PosProduct`). Always have a
  `factory <Name>.fromJson(Map<String, dynamic> j)`.

### Method / variable naming

- Methods, locals, parameters: `lowerCamelCase`.
- **Private members** (file-scope or class-scope) start with `_` —
  `_fetch`, `_storage`, `_extractError`. Use this freely; the analyser will
  warn on unused privates.
- Booleans read like predicates: `isLoading`, `isAppBlocked`, `hasToken`,
  `posOnline`.
- Async methods return `Future<T>` and are named as imperative verbs:
  `placeOrder`, `addUdhaar`, `refresh`, `lookupBarcode`.

### JSON ↔ Dart mapping

- Backend uses `snake_case` keys. Dart classes use `camelCase`.
- Conversion happens inside `fromJson` factories — **never** inside widgets or
  Notifier methods.
- Numeric fields from PostgreSQL can arrive as either `int`, `double`, or numeric
  string. The pattern used across the codebase is a local `extract(dynamic v)`
  helper that returns `double` — copy this pattern when adding a new model.

### Imports

- Group imports in three blocks, separated by a blank line, in this order:
  1. Dart SDK (`dart:async`, `dart:convert`)
  2. Third-party (`package:flutter/…`, `package:flutter_riverpod/…`, `package:http/…`)
  3. Project (`../../../core/services/api_client.dart`, etc.)
- Use **relative** imports inside the same feature, **package imports**
  (`package:kirana_ai/…`) only sparingly (a handful of files do this; do not
  spread further).

## 11. Local persistence

Two separate stores are used and they have different purposes — keep them that
way.

| Store                    | Keys                                                                          | Why                                        |
| ------------------------ | ----------------------------------------------------------------------------- | ------------------------------------------ |
| `flutter_secure_storage` | `auth_token`, `pos_token`                                                     | JWT material — encrypted on-device         |
| `SharedPreferences`      | `user_id`, `store_id`, `username`, `full_name`, `role`, `onboarding_completed`, `session_expiry` | Session metadata + cold-start routing flags |

`AuthRepository._saveSession()` writes both. `AuthRepository.clearSession()`
clears both **except** `onboarding_completed`. No other code should write the
auth keys directly.

## 12. Firebase

`lib/core/config/firebase_backend_config.dart` is the single point of Firebase
initialisation. It:

1. Initialises the Firebase app from `firebase_options.dart`.
2. Disables Crashlytics + Performance collection in debug builds.
3. Sets Remote Config defaults and a 1-hour minimum fetch interval in release
   (`Duration.zero` in debug for live updates).
4. Listens to `onConfigUpdated` and re-applies values on every push.

### Push notifications

Lives entirely under `lib/features/support/providers/notification_provider.dart`:

- `initLocalNotifications()` — called from `main.dart` before `runApp`. Registers
  the FCM background handler and the local-notification channel `kirana_ai_high`
  (matched by the meta-data tag in `AndroidManifest.xml`).
- `NotificationService.init()` — called once from `KiranaApp.build` and again
  from `DashboardScreen.initState` (idempotent). Sets up foreground + tap
  handlers.
- `NotificationService.uploadToken()` — POSTs the FCM token to
  `/kirana/auth/fcm-token`. Called after every successful login.

### Deep links

Configured in `android/app/src/main/AndroidManifest.xml`. Both
`https://lohiyaai.com/kiranaai/...` (App Links, requires SHA verification) and
the custom scheme `kiranaai://` resolve. The router strips the `/kiranaai`
prefix so deep-link paths line up with internal routes.

## 13. Error handling

- **HTTP errors** → `ApiException(statusCode, message)` thrown by `ApiClient` and
  `AuthRepository`. Catch them in the calling Notifier or screen and convert into
  user-facing messages.
- **Background / silent operations** (tracking, cart pings) wrap their HTTP call
  in `try { … } catch (_) {}` — explicitly swallowing because the user does not
  care if a tracking POST fails.
- **Uncaught Flutter errors** route to Crashlytics via `FlutterError.onError`.
- **Uncaught async errors** route to Crashlytics via
  `PlatformDispatcher.instance.onError`.
- Never `print()` in production code — the analyser will flag it. Use
  `debugPrint` for dev-only logging in code that is wrapped by `kDebugMode`.

## 14. Adding a new feature — checklist

The fastest reference is `features/baskets/` (small, recent, hits all the
patterns). Read it before starting.

1. **Pick a name.** Snake_case feature name; will become `lib/features/<name>/`.
2. **Scaffold folders:**
   - `lib/features/<name>/models/` (one file per model class).
   - `lib/features/<name>/providers/` (one provider per resource).
   - `lib/features/<name>/views/` (screens) and `views/widgets/` (local widgets).
3. **Define the data model.** Class + `fromJson` factory + (optional) `toJson`.
   Treat the JSON shape as the source of truth; do not "improve" key names
   client-side.
4. **Define the provider.** Pick one of the three Riverpod patterns from §7.b/c.
   - Read backend via `ref.read(apiClientProvider)`.
   - Expose `refresh()` for pull-to-refresh screens.
5. **Build the screen.** Use `ConsumerWidget` or `ConsumerStatefulWidget`. Watch
   the provider with `ref.watch(<feature>Provider)`. Render `AsyncValue.when`
   loading / error / data.
6. **Use shared widgets.** `PrimaryButton`, `BrandTextField`, `NotificationBell`.
   Pull colours from `BrandColors`.
7. **Wire navigation.** Add the route in `lib/app_router.dart`. Nest under
   `/profile` if it's a settings-style screen.
8. **Cross-invalidate.** If your writes affect another feature (e.g. customer or
   inventory), invalidate that provider.
9. **Gate Pro features.** Look up `subInfoProvider`/`subTierProvider`
   (`features/subscription/providers/subscription_provider.dart`). For tab-level
   gating, wrap with `_ProGateTab`; for action-level gating, call
   `showPaywallSheet(context, …)`.
10. **Add error handling.** Surface `ApiException.message` in a `SnackBar` or
    error banner; never blank-render an exception state.
11. **Test the cold path.** Run the app, sign out, sign back in, and make sure
    your screen loads from a fresh provider scope.

## 15. Lints and code style

The analyser is configured in `analysis_options.yaml` and includes
`package:flutter_lints/flutter.yaml`. No additional rules are enabled or
suppressed at the project level — keep it that way unless a rule is genuinely
broken.

Style summary that the lints enforce + that we enforce by convention:

- Prefer `const` constructors where possible.
- One import group per source: dart, package, project. No unused imports.
- No `print` — use `debugPrint` and gate noisy logs behind `kDebugMode`.
- Always provide a `Key? key`-less constructor: `const MyWidget({super.key, ...});`.
- Async functions return `Future<T>`. Do not mark synchronous code `async`.
- All public APIs (classes intended to be imported from another feature) should
  have at least a one-line `///` doc when their purpose isn't obvious from the
  name. Internal helpers do not need doc comments.

Run before pushing:

```bash
flutter analyze
flutter test
```

## 16. Known gotchas / "would have saved me a day"

- **`PosNotifier.build`** uses `Future.microtask(_loadProducts)` rather than
  awaiting in `build()` so Riverpod can complete `build` before the first state
  write. Do the same any time you need to kick off async work from a Notifier.
- **Hero tags must be unique.** The Overview tab uses `'overview_pos_fab'` and
  the POS/Inventory screen uses `'add_product_fab'`. If you add another FAB,
  give it its own `heroTag`.
- **POS Online badge** is computed reactively from
  `posState.error == null` (no async `initState` check). Do not regress to async
  state probing.
- **`AuthRepository.getCurrentUser`** falls back to the cached `SharedPreferences`
  copy if the network call fails. Without this, a flaky network would log users
  out. Preserve this fallback when refactoring.
- **`AppConfig` is a static singleton** that exposes `Stream<int>` for URL changes
  and `Stream<void>` for block-state changes. If you start watching a new
  Remote Config key, add it to `FirebaseBackendConfig._apply()` and a static
  field on `AppConfig` — do not read Remote Config directly from a widget.
- **Pro gating** is checked via `subInfoProvider.canAccessVendorManagement` /
  `canAccessCashflow` / `canAccessReferral`. Add a new accessor on
  `SubscriptionInfo` rather than hard-coding tier comparisons in widgets.
- **POS token can be `null`** for users who logged in via phone OTP but for
  whom `/pos/token-from-kirana` has not yet succeeded. Every `posGet`,
  `posPost`, `posGetList` returns `null` in that case — handle it.

## 17. Where to look first when something breaks

| Symptom                                              | Open first                                                            |
| ---------------------------------------------------- | --------------------------------------------------------------------- |
| Login spins forever / 401                            | `auth/repositories/auth_repository.dart`                              |
| Dashboard shows "Error loading"                      | `dashboard/providers/overview_provider.dart` (check `_fetch`)         |
| POS shows "Offline"                                  | `pos_inventory/providers/pos_provider.dart` (look for the `error` field) |

| FCM not arriving                                     | `support/providers/notification_provider.dart`                        |
| Trial countdown wrong / paywall behaviour            | `subscription/providers/subscription_provider.dart`                   |
| Crash on launch                                      | `main.dart` + `core/config/firebase_backend_config.dart`              |
| Wrong base URL on a new build                        | Firebase Console → Remote Config → `backend_url`                      |
